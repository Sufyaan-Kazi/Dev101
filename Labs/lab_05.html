Lab 5 - Connecting to a database

The Cities application was designed to presnet a list of cities back from a web service request In this lab, we’ll use Spring Boot to auto configure H2 for standalone mode and MySQL in Cloud Foundry.

Cloud Foundry services are managed through two primary types of operations:

Create/Delete

These operations create or delete instances of a service. For a database this could mean creating/deleting a schema in an existing multitenant cluster or creating/deleting a dedicated database cluster.

Bind/Unbind

These operations create or delete unique credential sets for an existing service instance that can then be injected into the environment of an application instance.

  2. Amend the City class to make it an Entity. Add the following annotations above the class declaration and annotate the fields below to replace the previous definition:

    @Entity
@Table(name = "uktowns")
public class City {

	@Id
	@GeneratedValue
	private long id;
	@Column
	private String name;
	@Column
	private String county;
	@Column(name = "gridref")
	private String stateCode;
	@Column(name = "postcode")
	private String postalCode;
	@Column
	private String latitude;
	@Column
	private String longitude;

  3. Create a new Java Interface called io.pivotal.fe.demos.repositories.CityRepository and make it look like below:

    package io.pivotal.fe.demos.repositories;

import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.data.repository.PagingAndSortingRepository;
import org.springframework.data.repository.query.Param;
import org.springframework.data.rest.core.annotation.RepositoryRestResource;
import org.springframework.data.rest.core.annotation.RestResource;

import io.pivotal.fe.demos.domain.City;

@RepositoryRestResource(collectionResourceRel = "cities", path = "cities")
public interface CityRepository extends PagingAndSortingRepository<City, Long> {

    Page<City> findAll(Pageable pageable);

    @RestResource(exported = false)
    City findById(Long id);

    @RestResource(path = "name", rel = "name")
    Page<City> findByNameIgnoreCase(@Param("q") String name, Pageable pageable);

    @RestResource(path = "nameContains", rel = "nameContains")
    Page<City> findByNameContainsIgnoreCase(@Param("q") String name, Pageable pageable);

    @RestResource(path = "state", rel = "state")
    Page<City> findByStateCodeIgnoreCase(@Param("q") String stateCode, Pageable pageable);

    @RestResource(path = "postalCode", rel = "postalCode")
    Page<City> findByPostalCode(@Param("q") String postalCode, Pageable pageable);
}

  4. Build and run the code again:

    $ ./gradlew build
$ ./gradlew bootRun

    5. Access the following URL: http://localhost:8080/cities and you will see a much bigger list of cities, formatted as neat JSON- yet we did not define any concrete Java Implementation.

    A couple of things are hapening here:

    1. Spring Boot has looked at our classpath and automatically constructed an instance of H2 and configured Spring data to access it.
    2. Secondly, we have pre-populated data in the databse using a product called Flyway (to find out more have a look here: Database Initialisation with Spring Boot. Spring Boot is automatically triggering Flyway and running it's initalisation.
    3. Finally, we have used Spring JPA to atuomatically create web service endpoints to access the data, with an ability to do wildcard searches, yet we have not written any actual implementation code.

    Try out these URLS and you will see how powerful this class is, without any direct implementation. Spring has guessed what we want to do simply bacause of the io.pivotal.fe.demos.repositories.CityRepository interface.

    http://localhost:8080/cities
http://localhost:8080/cities/search
http://localhost:8080/cities/search/name?q=Alfold
http://localhost:8080/cities/search/nameContains?q=Farn
http://localhost:8080/cities/search/nameContains?q=Fa&size=3

  5. Spring Boot made it really easy to expose this Spring JPA features - simply add the classes to the classpath and then annotate your class correctly.

    Spring Boot can also connect to traditonal databases such as MySQL should you have one on your computer or where your machine can access it.

    1. You would just need to comment out H2 and uncomment Mysql for example
    2. Add properties in application.yml for the db url, username, password etc
    3. Optionally, use Spring profiles to load different connection URL, username etc for different execution environments.
    4. Create an @Config class to read the db connection details and map these to params and initialise a db connection. Have a look at io.pivotal.fe.demos.config.DataSourceConfig.

The Services Marketplace

  1. When we push our application to cloud foundry, we may want to use a DB service provided in the cloud. Cloud Foundry exposes these services in the marketplace.

  2. There are two ways to discover what services are available on Pivotal Web Services. The first is available in the CLI. Just type:

    $ cf marketplace
$ cf marketplace -s p-mysql

    and you’ll get a list of services, their available plans and descriptions.

  3. The second way is specific to Pivotal Cloud Foundry’s Application Manager UI. If you haven’t already, login to it by visiting https://apps.emea-2.fe.gopivotal.com and click on the Marketplace link on the left.

Connecting to a MySQL Instance in Cloud Foundry

  1. Let’s begin by creating a MySql instance provided by Pivotal.

  2. From the CLI, let’s create a p-mysql service instance called 'MyDB':

    $ cf create-service p-mysql 100mb-dev MyDB

  3. Next we’ll push and bind the newly created instance to our cities application. First have a look at the manifest of this application, it's set to automatically bind to the 'MyDB' service - it doesn't define url, username etc at all. Now, deploy the application.

    $ ./gradlew build
$ cf push

    One of the libraries we uncommented in our build.gradle is FlyWay, we're using this to pre-initialise our databsase. You can watch in more detail how Flyway pre-loads data by opening a new window and tailing the logs or viewing the logs in the Apps Manager.

    Once deployed, access the application using the url supplied by CF (but append /cities to the end, e.g. https://your-URL/cities (Accept all warnings about SSL certificates).

    $ cf logs cities-service

  4. Let’s take a look at the environment variables for our application to see what’s been done. We can do this by typing:
